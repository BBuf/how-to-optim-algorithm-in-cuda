
## 0x0. 前言

相关帖子：

https://github.com/chardet/chardet/pull/322

https://github.com/chardet/chardet/issues/327

这两天 Python 圈里有一件事吵得很凶：老牌编码检测库 `chardet` 发布 `7.0.0` 之后，许可证从 `LGPL` 变成了 `MIT`，维护者还公开表示，这一版基本是借助 Claude Code 在几天里完成的 ground-up rewrite。

如果这只是一次普通的“老项目大重构”，事情未必会闹这么大。真正让它炸开的，是几个最敏感的词被一次性绑在了一起：**AI、重写、改许可证、继续沿用原项目身份**。

更关键的是，出来公开反对的不是普通围观用户，而是 `chardet` 的原作者 Mark Pilgrim 本人。

这件事有意思的地方也正在这里：表面上看是在吵一个 Python 库，往下挖其实会一路挖到 AI 生成代码、衍生作品认定、开源许可证边界，甚至开源社区最讲不清楚的“体面”问题。

## 0x1. 事情是怎么开始的？

时间线并不复杂，但每一步都踩在争议点上。

### 1. 3 月 2 日：PR #322 合并，`chardet 7.0` 登场

维护者 Dan Blanchard 提交并合并了 PR `#322`，标题非常直接：`chardet 7.0: ground-up MIT-licensed rewrite`。

在 PR 描述里，他给出了几个强烈的卖点：

- **许可证从 LGPL 改为 MIT**
- **保持与 5.x / 6.x API 兼容**
- **检测速度提升 27 倍**
- **准确率更高，支持更多编码**
- **测试数据迁出主仓库，以避免潜在许可证问题**
- **开发说明里明确写到：主要借助 Claude Code + superpowers，在不到一周时间内完成**

![](https://files.mdnice.com/user/59/70239728-5ffb-4f5f-9fc9-f2bd2f12cb8f.png)

这张图里最扎眼的几个词基本已经把后面的争议预告完了：`MIT-licensed rewrite`、`27x`、`Claude Code`、`less than a week`。站在发布者视角，这是一张很漂亮的成绩单；站在社区视角，这几乎等于把所有敏感点一次点亮了。

从工程宣传角度看，这几乎是一个“性能、准确率、许可证、AI 效率”四线齐飞的成功案例。

但真正的导火索，恰恰也是这四点叠在一起：

- 你说是“完全重写”，但项目名还是 `chardet`
- 你说是“MIT”，但旧项目长期是 `LGPL`
- 你说“没有基于 LGPL/GPL 代码”，但维护者自己维护旧项目十多年
- 你说“AI 重写”，那 AI 到底有没有见过旧代码？训练数据里有没有旧代码？

### 2. 3 月 4 日：原作者 Mark Pilgrim 亲自下场

`issue #327` 的标题是：`No right to relicense this project`。

Mark Pilgrim 在 issue 正文里说得非常明确：

- 自己是 `chardet` 原作者
- 当前维护者**无权**给项目重新授权
- LGPL 代码在被修改后，仍必须以 LGPL 发布
- 所谓“complete rewrite”并不自动构成免责
- 因为维护者长期接触原始代码，这不属于传统意义上的 clean-room implementation
- 要求项目恢复原有许可证

![](https://files.mdnice.com/user/59/2b306639-0cdc-49c1-ba51-6e6045aa514e.png)

这张图比后面很多长评论都更关键。因为从这里开始，事情已经不是“社区有人不舒服”，而是原作者本人直接出来说：你没有权利给这个项目换许可证。争议一下就从口水战升级成了带着明确权利主张的公开冲突。

这不是普通用户的道德谴责，而是原作者直接从版权与许可证角度提出反对。

也正因为这层身份，整个事件一下升级了。

## 0x2. 维护者的核心辩护是什么？

面对质疑，Dan Blanchard 后来给出了一段很长的回应，核心逻辑可以概括为四点。

### 1. “clean room 是手段，不是目的”

他承认，自己维护旧代码十多年，显然不满足传统 clean-room 中“知情人与实现者隔离”的要求。

但他强调，clean-room 方法论的目标是证明**新代码不是旧代码的衍生作品**，而不是必须机械满足某种流程仪式。换句话说，他认为“过程不够 clean”不等于“结果一定侵权”。

### 2. 用相似度工具证明“结构独立”

他提到使用 JPlag 对多个历史版本做相似度分析，并给出一个最关键的数字：

- `7.0.0 vs 6.0.0.post1` 的平均相似度约为 `0.04%`
- 最大相似度约为 `1.29%`

他的结论是：

- 过去一些版本虽然平均相似度不高，但最大相似度很高，说明仍有整块文件沿袭
- `7.0.0` 则没有任何文件在结构上延续旧实现
- 因此它不是“改写旧代码”，而是“独立作品”

原评论链接：<https://github.com/chardet/chardet/issues/327#issuecomment-4005195078>

![](https://files.mdnice.com/user/59/dbf78463-fba2-4033-86a7-448f6873e262.png)

![](https://files.mdnice.com/user/59/7d533d1b-5f74-4f7c-b782-f42efbebfa1e.png)

这两张图基本就是 Dan 整段回应里最硬的一部分。他的意思不是“我流程完全符合 clean room”，而是“就算流程不标准，结果也已经足够独立”。重点从“有没有严格隔离旧代码知识”，转到了“新代码最后还像不像旧代码”。

### 3. 重写过程是在空仓库里进行的

他公开描述的流程是：

- 先写设计目标
- 在空仓库启动重写
- 明确要求 Claude 不要基于 LGPL/GPL 代码
- 围绕 API 兼容、性能、准确率、无运行时依赖等目标不断迭代

但这里也埋下了后续最大的反证点：设计目标里明确提到“仍然叫 `chardet`”“目标是替代 chardet”“保持 API 兼容”“用 chardet 测试数据评估效果”。这让许多人认为，它并不是一个与原项目切断关系的新项目，而是一次针对原项目身份、生态位和用户基础的整体接管。

![](https://files.mdnice.com/user/59/aa6f33f5-2254-4e77-abca-2eaafc81dcb1.png)


![](https://files.mdnice.com/user/59/d1c06faf-ea13-4826-8263-da14ca278347.png)

把这两张图连起来看，会发现评论区其实不是单一方向地骂。一边是 Dan 把流程摊开，说自己在空仓库里做，并且明确要求 Claude 不要基于 LGPL/GPL 代码；另一边则是 `mitsuhiko` 这类人公开表示“目前看起来像 clean-room implementation”，紧接着又有人贴出 agent 计划里参考旧文件的证据。也就是说，这场争论并不是一边倒，而是双方都能拿出自己觉得站得住的材料。

### 4. 为什么想改成 MIT？

这也是争议里一个很重要、但一开始没有说清楚的部分。

Dan 后来解释：

- 自己多年前就希望把 `chardet` 变成更宽松许可证
- 曾经有人讨论过把它纳入 Python 标准库，但标准库通常要求更宽松的许可证，比如 MIT/BSD
- 他还希望通过更宽松的许可证吸引更多贡献者，摆脱“一人维护项目”的状态

![](https://files.mdnice.com/user/59/9bc3e70c-71f4-4f6f-abda-e22a0dd74859.png)

这一段也很关键。因为如果没有这张图，很多人会以为这次改许可证只是为了配合 AI 重写顺手做的。但从维护者自己的说法看，想把 `chardet` 换成更宽松许可证这件事，其实已经酝酿很多年了。AI 更像是把一件拖了很久的事，突然推进到了能落地的程度。

也就是说，在他的叙述里，“AI 重写”只是手段，“改成 MIT”其实是一个酝酿多年的目标。

## 0x3. 社区为什么反应这么激烈？

如果只看代码重写本身，社区未必会炸成这样。真正让很多人愤怒的，是下面这几个点叠加在一起。

### 1. 大家反对的，不只是“AI 写代码”，而是“AI 帮你洗许可证”

这是争论中最刺耳、也最常见的一种指控：

> 这看起来像是在把一个 LGPL 项目，借助 LLM 重新表述一遍，然后洗成 MIT。

很多评论者并不否认“理论上存在合法独立重写”的可能，但他们认为：

- 维护者长期接触旧代码
- LLM 很可能训练过旧代码
- agent 在开发过程中可能还读取过旧仓库内容
- 新版本继续沿用原项目名、原分发渠道、原社区信誉

在这样的前提下，再宣称“这已经不是 LGPL 项目了”，在很多人看来，几乎就是对 copyleft 精神的公开规避。

### 2. “你完全可以新建一个项目，为什么非要占用原项目身份？”

这是讨论里支持度非常高的一条观点。

很多人并不一定完全断言“MIT 重写必然违法”，但他们认为最起码有一件事是显而易见的：

- 如果你真写出了一个全新的、许可证无污染的实现
- 最稳妥、最体面的做法，是另起一个名字，比如 `chardet-ng`
- 让用户自行选择是否迁移

而不是直接把原仓库、原项目名、原 issue 历史、原生态影响力，一起承接过来，再宣布“从这一刻起它是 MIT 了”。

在很多评论者看来，这一步即使暂且不谈法律，至少已经突破了开源社区默认的伦理边界。

![](https://files.mdnice.com/user/59/654100d4-77a5-488e-84ba-bd91465e164e.png)

这张图里有两层意思。上半部分有人直接拿美国版权局关于 AI 可版权性的报告来怼维护者，意思是“如果代码不是你亲手写的，你到底贡献了什么”；下半部分则回到了更务实的社区方案：就算你这套说法部分成立，最起码也应该作为一个单独项目发布，而不是直接挂到旧项目名下。

### 3. 这被视为一种供应链风险

PR #322 的评论里已经有人把它描述为一种新的 software supply chain risk。

原因并不难理解：

- `chardet` 是 Python 生态里历史非常久、依赖面很广的组件
- 下游用户通常默认“这个包延续了历史项目的法律与质量边界”
- 但这次发布同时改了**实现来源、许可证基础、治理叙事**

对于下游团队来说，真正可怕的不只是性能变化，而是：

- 现在这个 MIT 声明到底稳不稳？
- 如果未来被认定存在版权问题，下游是否会被牵连？
- 如果 AI 生成代码本身版权归属不清，许可证声明是否站得住？

当一个高传播依赖包把这些不确定性一次打包带进生态时，很多公司法务和工程团队都会本能警惕。

![](https://files.mdnice.com/user/59/5956222a-a1f9-4558-9632-185c85abcd86.png)

这张图里的关键词很直接：`The sensible thing`、`chardet-ng`、`supply chain risk`。评论者的意思不是简单喊一句“回滚”，而是说如果这些优点都是真的，那完全可以放到一个新名字下面去做；真正让人不安的，不是出现了一个 vibe-coded 的新库，而是它被当作原来的 `chardet` 新版本推到了依赖链面前。

## 0x4. 争论最核心的几个法律问题

整场讨论其实反复围绕四个问题打转，而且直到现在都没有定论。

### 1. AI 重写出来的代码，到底是不是旧项目的衍生作品？

反对者认为：

- 维护者本人长期接触旧代码，已经无法满足 clean-room 条件
- 即便提示词里说“不要参考 LGPL 代码”，也无法证明模型真的没受影响
- 如果模型训练集中本来就有 `chardet`，那输出与旧项目之间的关系更加复杂

支持者则认为：

- 版权保护的是表达，不是思想、算法和接口目标本身
- 只要新实现没有继承原代码表达，就有可能构成独立作品
- 相似度极低、文件结构完全不同，是一个重要证据

换句话说，双方争的是：**“独立实现”的认定，究竟更看过程，还是更看结果。**

### 2. 如果真是 AI 生成代码，这些代码本身能不能被版权保护？

这也是评论区另一条高频线索。

一种观点认为：

- 美国版权局对纯 AI 生成内容的可版权性持非常谨慎态度
- 如果代码主要由 LLM 生成，而人类创作性不足，那它可能根本不具备可版权性
- 既然没有版权，就谈不上“以 MIT 许可它”

另一种观点则认为：

- 现实中的 agentic coding 往往不是纯自动生成
- 人类在设计、筛选、迭代、测试、组合上的参与，可能足以构成可版权部分
- 因此不能简单说“LLM 产物就一定属于公有领域”

这个问题放在整个 AI 软件行业里都没有完全定型。`chardet` 只是把它提前暴露出来了。

### 3. API 兼容、数据映射、测试套件，算不算“污染”证据？

这是支持 MIT 一方最容易低估、反对一方最抓得住的一个点。

在 issue 评论里，有人翻出了设计文档中的内容，显示 agent 至少在某些环节被要求：

- 与 `chardet 6.x` API 兼容
- 继续命名为 `chardet`
- 使用 `chardet` 的测试体系或参考数据
- 某些映射信息甚至被要求参考旧版文件

于是问题变成：

- 即便代码 token 不像了
- 但如果关键数据组织、元信息映射、行为约束都来自旧项目
- 这还能不能算“完全独立”？

这也是为什么很多人对单纯依赖 JPlag 分数并不买账。因为他们觉得，相似度工具能证明“没明显抄代码”，却不能自动证明“没有继承受版权保护的表达结构”。

![](https://files.mdnice.com/user/59/f19a9784-925b-43ff-9a1d-00bcc98150e1.png)

这张图很典型地体现了评论区的法律分歧：有人拿 `Google v. Oracle` 讨论 API 兼容重写在美国法下到底怎么认定，也有人马上回到更朴素的判断，说重点根本不在 API，而在于这次重写是不是仍然“heavily based on the original code”。说白了，大家连“该讨论什么问题”都没有完全达成一致。

### 4. 许可证之外，项目名和项目身份算不算另一层问题？

还有一条有意思的讨论是：哪怕暂时先不谈代码版权，只看项目身份，原仓库、原名字、原分发渠道本身是不是一种不可忽视的“资产承接”？

有评论者提到，这可能不只是 copyright 问题，也可能涉及 trademark、社区信誉继承、用户预期误导等更广义的治理问题。

简单说就是：

- 代码能否合法重写，是一个层面
- 但“重写完以后还能不能理直气壮说这就是原项目的新版本”，是另一个层面

## 0x5. 舆论场上最有代表性的三种立场

如果把几百条评论压缩一下，大致可以分成三派。

### 1. 强烈反对派：这就是违规 relicensing

这派人的看法最直接：

- 原项目是 LGPL
- 当前版本仍然以同一项目身份延续
- 维护者没有获得所有版权持有者同意
- 所谓 AI 重写并不能神奇消灭衍生关系

因此他们主张：

- 立即恢复 LGPL
- 或者至少把 MIT 版本另起仓库、另起名字
- 必要时由原作者或相关机构推动法律行动

### 2. 法律观望派：道理先别说满，法院都未必说得清

这派更强调不确定性：

- 版权法在 AI 代码生成时代本来就没完全准备好
- “是否构成衍生作品”往往要看司法辖区和具体证据
- 社区里很多断言其实都超出了大家真正掌握的信息

他们通常不会为这次操作背书，但会提醒：

- 现在大多数结论都还是推测
- 真要落地，可能只能靠法官、法务、甚至未来判例来回答

### 3. 伦理批评派：先不谈法，光从做人做事就不对

这派评论在情绪上往往最有共鸣。

他们的重点不是逐条解释 LGPL，而是说：

- 你可以觉得自己在法律上站得住
- 但你不能忽视原作者、老贡献者和用户会感到被背刺
- 开源不只是许可证文本，也是长期形成的社会契约

![](https://files.mdnice.com/user/59/9bbf085c-6722-47e2-af04-7ee7c6532c80.png)

![](https://files.mdnice.com/user/59/3edb4aab-2fc2-42ea-a474-b124450bd3fb.png)

这两张图把评论区里情绪层面的东西表达得很直接：有些人已经不太关心法条怎么解释了，他们更在意的是，这件事在开源社区的语境里看起来就是“不太对劲”。这类评论未必最严谨，但它们解释了为什么这场争论会迅速失控，因为很多人讨论的其实不是同一个问题：有人在讲版权，有人在讲规则，有人在讲体面。

一句话概括就是：

> 也许你能争出“合法”，但很难争出“体面”。

## 0x6. 为什么这件事会成为一个样本案件？

`chardet` 之所以特殊，不在于它一定会成为法律判例，而在于它几乎把几个最敏感的问题一次性叠在了一起：

- **老牌开源项目**
- **原作者仍在且明确反对**
- **维护者长期接触旧代码**
- **AI/agent 参与重写**
- **许可证从 copyleft 变成 permissive**
- **项目名、包名、生态位置全部延续**

如果未来开源世界还会出现类似操作，那么几乎每个项目都要回答下面这些问题：

- 用 AI 做“独立重写”，边界到底在哪？
- 代码相似度低，是否就足够证明不是衍生作品？
- 模型训练数据中的旧项目，会不会破坏 clean-room 主张？
- 维护者能否借重写机会，单方面改写项目的许可证与身份？
- 下游使用者该如何评估这类项目的法律与供应链风险？

这些问题今天都还没有标准答案，`chardet` 只是把它们提前暴露出来了。现在看它像一场热闹，再过一段时间回头看，它大概率会被当成一个典型样本反复提起。

## 0x7. 目前最现实的结局会是什么？

从现在公开信息看，短期内可能出现三种路径：

### 1. 维持现状，继续争论

这是最容易发生的情况。

维护者坚持 MIT 合法，反对者继续认为这侵犯了 LGPL 或至少违背开源伦理。项目继续存在，但争议标签会长期跟着它。

### 2. 社区分叉

一些用户回退到 `6.x`，或者围绕旧 LGPL 版本继续 fork。这其实已经是很多人给出的务实建议：

- 需要法律风险更低的，就继续用旧版本
- 不接受新治理方式的，就另起维护分支

### 3. 被迫重新定义边界

如果未来真出现正式法律行动、平台治理介入、或更明确的版权判例，那么 `chardet` 很可能会被当成一个参考案例。

到那时，大家争的可能就不只是这个项目本身，而是：

- AI 代码生成到底如何适配现有开源法理
- copyleft 在 agent 时代是否会被系统性绕开
- 开源社区要不要为“AI 重写接管老项目”建立新的共识与规范

## 0x8. 总结

如果只从工程结果看，这次发布其实很诱人：更快、更准、API 还兼容，甚至连“许可证更宽松”都能被包装成一个利好消息。

但 `chardet` 这场风波真正让人不安的地方，不在于它到底是不是一次成功重写，而在于它把一个过去还算模糊的问题直接摆到了桌上：

如果以后越来越多的老项目都可以用“AI 重写 + 相似度很低 + 继续沿用原项目身份”的方式，去完成一次事实上的再定义，那原作者意志、历史许可证、贡献者预期这些东西，边界到底还剩多少？

这件事今天当然没有答案。短期内它大概率还是会停留在“有人觉得合理，有人觉得离谱”的状态里继续吵下去。但再往后看，它真正留下来的问题可能不是 `chardet` 自己，而是开源社区以后要怎么面对这类越来越多、也越来越难判的案例。

最后想留下的不是结论，而是一个问题：

**当 AI 已经足以把一个老项目“重写到看起来不像原项目”，开源社区还能靠什么去维护原来那套默认成立的边界？**
